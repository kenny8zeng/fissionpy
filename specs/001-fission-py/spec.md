# Feature Specification: Python 单体文件裂变拆分工具

**Feature Branch**: `001-fission-py`
**Created**: 2026-05-10
**Status**: Draft
**Input**: 用户需要一个CLI工具，将过大的Python单体文件按模块相关性拆分为多个py模块文件。工具基于LibCST分析结构，依赖信息存入SQLite，支持树形视图、YAML拆分计划、代码提取、逻辑校验、依赖更新。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 分析单体文件结构与依赖 (Priority: P1)

开发者有一个过大的 Python 单体文件，需要理解其内部结构。CLI 工具解析该文件，识别所有顶层符号（函数、类、变量），分析它们之间的依赖关系，将结果写入 SQLite 数据库，并可在终端以树形视图展示符号及依赖层级。

**Why this priority**: 分析是所有后续操作的基础，没有结构信息就无法做任何拆分决策。

**Independent Test**: 给定一个包含多个函数和类的 Python 文件，执行分析命令后，SQLite 数据库中应包含完整的符号表和依赖关系表，树形视图输出应清晰展示层级。

**Acceptance Scenarios**:

1. **Given** 一个包含 3 个顶层函数和 2 个顶层类的 Python 文件，**When** 开发者运行分析命令，**Then** SQLite 数据库中记录 5 个顶层符号及其类型、行号范围
2. **Given** 函数 A 调用了函数 B，**When** 分析完成后查看依赖表，**Then** 依赖表中记录 A 依赖 B 的关系
3. **Given** 分析已完成，**When** 开发者运行树形视图命令，**Then** 终端以缩进树形格式展示每个符号及其直接依赖
4. **Given** 一个空文件或仅含 import 的文件，**When** 运行分析命令，**Then** 工具正常完成，符号表为空，不报错

---

### User Story 2 - 生成与编辑拆分计划 (Priority: P2)

开发者在查看分析结果后，需要决定哪些符号提取到哪个模块。CLI 工具根据分析结果生成 YAML 格式的拆分计划模板，列出所有顶层符号，默认全部保留在主文件中。开发者编辑该模板，标记需要提取的符号及目标模块名。

**Why this priority**: 计划生成依赖分析结果，且是执行提取的前提。没有计划就无法执行拆分。

**Independent Test**: 给定已分析的文件，运行计划生成命令后输出 YAML 文件，所有符号默认标记为保留，用户修改后可被工具正确解析。

**Acceptance Scenarios**:

1. **Given** 分析结果中有 5 个顶层符号，**When** 运行计划生成命令，**Then** 生成 YAML 文件包含所有 5 个符号，每个符号默认标记为 `retain`（保留在主文件）
2. **Given** 用户将符号 A 标记为提取到模块 `utils`，**When** 工具重新读取该 YAML，**Then** 工具正确识别 A 需提取到 utils 模块
3. **Given** YAML 计划中引用了不存在的符号名，**When** 工具校验计划，**Then** 报告错误并指出无效符号名

---

### User Story 3 - 执行代码提取与进度追踪 (Priority: P3)

开发者确认拆分计划后，CLI 工具按计划将指定符号从单体文件提取到新模块文件。提取过程中实时将进度和元数据记录到 SQLite（每个符号的提取状态、目标模块、提取时间等）。提取完成后，根据提取元数据自动更新受影响模块的 import 语句，确保依赖链路完整。

**Why this priority**: 提取是核心操作，但依赖分析和计划生成必须先完成。实时进度追踪和元数据驱动的 import 更新是保证拆分正确性的关键。

**Independent Test**: 给定有效的 YAML 计划，运行提取命令后，指定符号被写入对应模块文件，SQLite 中记录提取状态，import 语句被正确更新。

**Acceptance Scenarios**:

1. **Given** 计划要求将函数 A 和类 B 提取到模块 `utils`，**When** 执行提取，**Then** 生成 `utils.py` 包含 A 和 B，且 SQLite 中 A 和 B 的提取状态为 `extracted`
2. **Given** 提取过程中符号 A 已完成但符号 B 失败，**When** 查看进度，**Then** SQLite 中 A 状态为 `extracted`，B 状态为 `failed`，已完成部分不回滚
3. **Given** 主文件中函数 C 依赖了已提取到 `utils` 模块的函数 A，**When** 提取完成后更新依赖，**Then** 主文件自动添加 `from utils import A`
4. **Given** 新模块 `utils` 中的函数 A 依赖了仍留在主文件的函数 D，**When** 提取完成后更新依赖，**Then** `utils.py` 自动添加 `from main_module import D`（或按包结构正确的 import 路径）

---

### User Story 4 - 校验提取结果与重组主文件 (Priority: P4)

提取完成后，工具校对提取后的内容与原始内容的逻辑一致性。校验通过后，将原始单体文件重命名为备份文件，然后将剩余未提取的代码块写入新的同名主文件，确保主文件只包含保留的符号及必要的 import。

**Why this priority**: 校验和重组是最后的收尾步骤，确保拆分结果安全可靠。如果前面的步骤都已正确完成，这一步是质量保障。

**Independent Test**: 给定已完成提取的状态，运行校验和重组命令后，原始文件被备份，新主文件仅含保留符号，所有模块可被 Python 正常导入。

**Acceptance Scenarios**:

1. **Given** 提取已完成，**When** 运行校验命令，**Then** 工具对比原始 AST 与拆分后所有模块的合并 AST，报告是否逻辑一致
2. **Given** 校验发现符号缺失或逻辑不一致，**When** 输出校验结果，**Then** 明确列出缺失的符号名和位置
3. **Given** 校验通过，**When** 执行重组，**Then** 原始文件被重命名为 `<原名>.bak`，新主文件仅含保留符号及必要的 import 语句
4. **Given** 拆分后的项目目录，**When** Python 尝试 import 主文件，**Then** 无 ImportError，所有符号可通过正确路径访问

---

### Edge Cases

- 目标单体文件包含条件性导入（`try/except import`）时，依赖分析如何处理？
- 顶层符号存在循环依赖时，提取到不同模块可能导致循环 import，工具应如何检测和警告？
- 如果 YAML 计划将所有符号都标记为提取，主文件应保留什么？（至少保留 `__init__.py` 风格的重导出存根）
- 同名模块文件已存在时，提取是否覆盖还是报错？
- 符号的装饰器中引用了其他顶层符号，依赖分析是否覆盖装饰器依赖？
- 文件中包含 `if __name__ == "__main__"` 块，该块不属于任何顶层符号但可能依赖多个符号，如何处理？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 工具 MUST 解析指定 Python 单体文件，识别所有顶层符号（函数定义、类定义、变量赋值语句）
- **FR-002**: 工具 MUST 分析顶层符号之间的引用依赖关系，包括函数调用、类实例化、变量引用、装饰器引用、类型注解引用
- **FR-003**: 分析结果 MUST 写入 SQLite 数据库，包含符号表（名称、类型、行号范围、源码文本）和依赖表（依赖方、被依赖方、依赖类型）
- **FR-004**: 工具 MUST 提供树形视图打印功能，以缩进层级展示每个顶层符号及其直接依赖符号
- **FR-005**: 工具 MUST 根据 SQLite 中的分析结果生成 YAML 格式的拆分计划模板
- **FR-006**: YAML 模板中 MUST 列出所有顶层符号，每个符号默认标记为保留在主文件（`retain`），用户可将其修改为指定目标模块名
- **FR-007**: 工具 MUST 支持用户编辑后的 YAML 计划校验，检查符号名有效性、目标模块名合法性
- **FR-008**: 根据校验通过的 YAML 计划，工具 MUST 将指定符号的代码块无损提取到对应的新模块文件中——保留原始注释、空行、缩进和代码格式，不丢失任何非代码信息
- **FR-009**: 提取过程 MUST 实时将每个符号的提取进度和元数据写入 SQLite（提取状态、目标模块、提取时间戳、源行号范围）
- **FR-010**: 提取完成后，工具 MUST 根据提取元数据自动更新所有受影响模块的 import 语句，确保跨模块依赖链路完整
- **FR-011**: 工具 MUST 对提取结果与原始内容进行逻辑一致性校验，检测符号缺失或语义变化
- **FR-016**: 工具 MUST 校验提取后的代码块与原始代码在文本层面一致（注释、空行、缩进无丢失），确保格式无损
- **FR-012**: 校验通过后，工具 MUST 将原始单体文件重命名为备份文件（添加 `.bak` 后缀）
- **FR-013**: 工具 MUST 将剩余未提取的代码块从备份文件写入新的同名主文件，包含必要的 import 语句和保留符号的完整代码
- **FR-014**: 工具 MUST 在主文件中为被其他模块依赖的已提取符号添加重导出 import（如 `from module import Symbol`），保持向后兼容性
- **FR-015**: 工具 MUST 检测循环依赖风险（两个或多个模块互相 import），并发出警告

### Key Entities

- **顶层符号（TopLevelSymbol）**: Python 文件中定义在模块顶层的函数、类、变量；属性包括名称、类型、源码行号范围、源码文本
- **依赖关系（Dependency）**: 两个顶层符号之间的引用关系；属性包括依赖方符号、被依赖方符号、依赖类型（调用/引用/继承/装饰器）
- **拆分计划（ExtractionPlan）**: YAML 格式的符号提取指令集合；每个符号对应一条指令（保留或提取到指定模块）
- **提取元数据（ExtractionMeta）**: 记录提取过程中每个符号的状态信息；属性包括提取状态、目标模块、时间戳、源行号范围
- **模块文件（ModuleFile）**: 提取后生成的新 Python 文件；包含提取的符号代码及自动生成的 import 语句

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 1000 行以内的 Python 单体文件，结构分析和依赖分析完成时间不超过 5 秒
- **SC-002**: 树形视图清晰展示符号层级和依赖关系，开发者无需查看源码即可理解文件结构
- **SC-003**: 拆分后的所有模块文件可被 Python 解释器无错误导入（无 ImportError、无 AttributeError）
- **SC-004**: 校验功能能 100% 检测出提取前后顶层符号的缺失
- **SC-004a**: 提取后的代码块与原始代码文本逐行一致，注释和空行零丢失
- **SC-005**: 提取过程中断后重启，可通过 SQLite 中的进度元数据恢复或了解中断位置
- **SC-006**: 自动生成的 import 语句覆盖所有跨模块依赖，开发者无需手动修补即可运行拆分后的项目

## Design Constraints

- **代码无损提取**: 选择 LibCST 作为解析引擎的核心原因是其能够完整保留注释、空行、缩进和代码格式。与标准 `ast` 模块不同，LibCST 保留所有空白和注释信息，确保提取后的代码块与原始文件中的外观完全一致——包括函数/类上方的文档注释、行内注释、装饰器注释、间隔空行等均不丢失

## Assumptions

- 目标单体文件是合法的 Python 代码，无语法错误
- 顶层符号的粒度为整个函数/类/变量定义，不考虑将函数内部逻辑拆分到不同模块
- 拆分后的模块文件与原始文件位于同一目录（同一 Python 包内）
- `if __name__ == "__main__"` 块始终保留在主文件中
- 条件性导入（`try/except import`）的依赖分析按静态最大集处理（即假设所有分支都可能执行）
- 备份文件使用 `.bak` 后缀，不会自动删除，由开发者决定是否清理
- 工具仅处理单文件拆分场景，不涉及跨多个文件的代码迁移
