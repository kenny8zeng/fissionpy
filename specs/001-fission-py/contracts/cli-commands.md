# CLI Command Contracts: fissionpy

**Feature**: 001-fission-py
**Date**: 2026-05-10

## 命令总览

| 命令 | 用途 | 输入 | 输出 |
|------|------|------|------|
| `fission analyze` | 解析单体文件，写入 SQLite | 文件路径 | SQLite 数据库 |
| `fission tree` | 打印符号依赖树 | SQLite 数据库 | 终端树形视图 |
| `fission plan` | 生成 YAML 拆分计划模板 | SQLite 数据库 | YAML 文件 |
| `fission extract` | 执行代码提取 | YAML 计划文件 | 新模块文件 + SQLite 进度 |
| `fission verify` | 校验提取结果 | 原文件 + 拆分后目录 | 校验报告 |
| `fission reorganize` | 备份原文件 + 重组主文件 | SQLite + 备份文件 | 新主文件 |

---

## `fission analyze`

### 输入

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `file` | PATH | 是 | 目标 Python 单体文件路径 |
| `--db` | PATH | 否 | SQLite 数据库路径，默认 `./fission.db` |
| `--force` | FLAG | 否 | 强制重新解析（即使 hash 未变） |

### 输出

- SQLite 数据库写入 `files`/`symbols`/`dependencies` 三张表
- stdout: 解析摘要（符号数、依赖数、耗时）

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 解析成功 |
| 1 | 文件不存在或不可读 |
| 2 | 语法解析失败（非合法 Python） |

## `fission tree`

### 输入

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `--db` | PATH | 否 | SQLite 数据库路径，默认 `./fission.db` |
| `--symbol` | TEXT | 否 | 仅展示指定符号的子树 |
| `--reverse` | FLAG | 否 | 反向视图：展示谁依赖了该符号 |

### 输出

- 终端树形视图（rich.tree 渲染）
- 颜色编码：函数=青色，类=黄色，变量=白色，依赖箭头=灰色

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 展示成功 |
| 1 | 数据库不存在或无分析数据 |

## `fission plan`

### 输入

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `--db` | PATH | 否 | SQLite 数据库路径，默认 `./fission.db` |

## `fission extract`

### 输入

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `plan` | PATH | 是 | YAML 拆分计划文件路径 |
| `--db` | PATH | 否 | SQLite 数据库路径，默认 `./fission.db` |
| `--resume` | FLAG | 否 | 从上次中断处继续提取 |

### 处理流程

1. 校验 YAML 计划合法性
2. 逐符号执行提取，每个符号更新 extraction_progress 状态
3. 根据提取元数据更新所有模块的 import 语句
4. 检测循环依赖风险

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 全部提取成功 |
| 1 | 计划校验失败 |
| 2 | 部分符号提取失败（已提取部分保留） |
| 3 | 检测到循环依赖风险 |

## `fission verify`

### 输入

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `--db` | PATH | 否 | SQLite 数据库路径，默认 `./fission.db` |
| `--source` | PATH | 否 | 原始单体文件路径（从 YAML 计划读取默认值） |

### 校验项

| 校验项 | 对应需求 | 说明 |
|--------|---------|------|
| 符号完整性 | FR-011 | 所有原始符号在拆分后文件中可找到 |
| 格式无损性 | FR-016 | 提取后代码与原始代码文本逐行一致 |
| import 完整性 | FR-010 | 所有跨模块依赖有对应 import 语句 |

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 全部校验通过 |
| 1 | 校验发现差异 |

---

## `fission reorganize`

### 输入

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `--db` | PATH | 否 | SQLite 数据库路径，默认 `./fission.db` |
| `--plan` | PATH | 否 | YAML 计划文件，默认 `./fission-plan.yaml` |

### 处理流程

1. 确认 verify 已通过（否则拒绝执行）
2. 原始单体文件重命名为 `<原名>.bak`
3. 从备份文件提取保留符号写入新同名主文件
4. 添加重导出 import（FR-014）

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 重组成功 |
| 1 | verify 未通过或未执行 |
| 2 | 备份文件已存在（需手动处理） |

## 全局选项

| 选项 | 类型 | 说明 |
|------|------|------|
| `--db` | PATH | SQLite 数据库路径，所有命令共享，默认 `./fission.db` |
| `--verbose` | FLAG | 详细输出模式 |
| `--version` | FLAG | 显示版本号 |
